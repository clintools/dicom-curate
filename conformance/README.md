# dciodvfy conformance

Checks that **curation does not make DICOM header conformance worse** according to `dciodvfy -new` output ‚Äî not that files are standards-valid.

## Overview of files

| Path | Role |
|------|------|
| `dciodvfy.ts` | Run, parse, normalise, subset check |
| `baseline.ts` | Load and compare baseline JSON |
| `helpers.ts` | Ephemeral synthetic fixtures, passthrough curate |
| `public-cases.json` | Public dirty corpus catalog: sha256-pinned quirky files fetched from upstream (see [Public dirty corpus](#public-dirty-corpus)) |
| `publicCases.ts` | Loads `public-cases.json` |
| `localFixtures.ts` | `CONFORMANCE_LOCAL_PATH` discovery and baseline paths for personalised test datasets |
| `differentialSuite.ts` | Shared four-test dciodvfy differential registration |
| `allowlist.ts` | Suppressed violation patterns |
| `baselines/` | Committed expected `dciodvfy` output (akin to 'snapshots') |
| `*.test.ts` | Vitest (`conformance` project) |

## What we are testing

| Question | Answer |
|----------|--------|
| Is the file valid DICOM for clinical use? | **No** ‚Äî many fixtures are intentionally imperfect. |
| Does `curateOne` add new `dciodvfy` findings? | **Yes** ‚Äî that is the regression gate. |
| Does output match a pinned `dciodvfy` snapshot? | **Partly** ‚Äî see baseline drift tests below. |

Flow for the main check:

1. Run `dciodvfy` on input (synthetic file from `dicom-synth`, a fetched public case, or **your** local `.dcm` via env ‚Äî see [Local and private fixtures section](#local-and-private-fixtures)).
2. Run passthrough `curateOne` (mapping changes nothing; file is still rewritten).
3. Run `dciodvfy` on output.
4. Fail if any **new** normalised violation appears versus the input (or versus a committed baseline).

We do **not** require violations to disappear after curate. We only require the violation **set** not to grow.

## How to read results

| Outcome | Meaning |
|---------|---------|
| **Pass** | No new violations introduced (and, where applicable, live output still matches the committed baseline). |
| **Fail ‚Äî introduced** | Curate (or the test setup) added at least one normalised `dciodvfy` line not present before. Printed as `introduced` in the assertion message. |
| **Fail ‚Äî baseline drift** | Live `dciodvfy` on the fixture differs from the JSON in `baselines/` (extra or missing lines). Usually means `dicom3tools` was upgraded, or `dicom-synth` output changed. Refresh baselines after confirming the change is expected. |
| **Skipped** | `dciodvfy` not on `PATH` and `DCIODVFY_PATH` unset ‚Äî install [dicom3tools](https://www.dclunie.com/dicom3tools.html) or set `DCIODVFY_PATH`. |
| **Skipped (public)** | `RUN_PUBLIC_CONFORMANCE` unset (or `0`/`false`) ‚Äî needs network on first run (fetched files are cached). CI sets it; local runs opt in. |
| **Skipped (local)** | `CONFORMANCE_LOCAL_PATH` unset ‚Äî optional private corpus; off in default CI. |
| **Fail (local config)** | `CONFORMANCE_LOCAL_PATH` set but invalid (missing path, no `.dcm` files, etc.) ‚Äî one failing test reports the error. |

Normalised violation keys look like `Error::</TagName(gggg,eeee)>::message` (file paths stripped). See `dciodvfy.ts` ‚Üí `normaliseViolation`.

## The `baselines/` folder

**Not DICOM files.** Each `*.dciodvfy-baseline.json` is a saved list of normalised violation strings from one `dciodvfy -new` run on a fixture **at a point in time**.

```json
{
  "label": "minimal-ct-0.dcm",
  "violations": [
    "Error::</StudyDate(0008,0020)>::Missing attribute ‚Ä¶",
    "‚Ä¶"
  ],
  "notes": "Regenerate with pnpm update:conformance-baselines"
}
```

| Directory | File | Source at test time |
|-----------|------|---------------------|
| `baselines/synthetic/` | `minimal-ct-0`, `valid-uid-ct-0`, `vendor-warnings-ct-0` | Generated by `dicom-synth` into a temp dir (not committed) |
| `baselines/public/` | One per non-`dciodvfy_skip` case in `public-cases.json` | Fetched from the pinned URL when the public tests run |

**Why commit JSON but not `.dcm`?** Baselines pin **validator output**, not bytes on disk. Synthetic DICOM is reproduced from `dicom-synth`; only the expected `dciodvfy` lines are versioned.

**When to update:** After upgrading `dicom3tools`, or after an intentional change to `dicom-synth` fixtures. Run `pnpm update:conformance-baselines` and commit the JSON diff.

## Test files

| File | Tests | Default CI |
|------|-------|------------|
| `dciodvfy.differential.test.ts` | Per synthetic variant: baseline drift; identity; byte copy; passthrough curate | Yes |
| `dciodvfy.normalise.test.ts` | Parser normalisation only (no `dciodvfy` binary) | Yes |
| `dciodvfy.public.test.ts` | Catalog validation + baseline presence (always); shared four-test differential suite per public case after a `beforeAll` prefetch (`RUN_PUBLIC_CONFORMANCE=1`) | Yes (env set in workflow) |
| `dciodvfy.local.test.ts` | Optional local file or directory (env) | Local test: no |
| `localFixtures.test.ts` | Path discovery helpers (no `dciodvfy` binary) | Yes |

### `dciodvfy.differential.test.ts` (per fixture)

1. **live before matches committed baseline** ‚Äî Today's `dciodvfy` on the synth file equals `baselines/synthetic/<id>.json`. Catches toolchain upgrades without a deliberate baseline refresh.
2. **same file has empty introduced set** ‚Äî Sanity: comparing a file to itself introduces nothing.
3. **byte-identical copy introduces no new violations** ‚Äî Control: copy file, re-run `dciodvfy`; path in output may differ but violation set should not grow.
4. **passthrough curate introduces no new dciodvfy violations** ‚Äî **Primary regression test** for `curateOne`. Fails if rewrite adds findings versus live input; when a baseline JSON exists on disk, also checks versus that snapshot.

### `dciodvfy.normalise.test.ts`

Unit tests for line parsing and normalisation. Always runs; does not need `dciodvfy`.

### `dciodvfy.public.test.ts`

1. **catalog validation** (always) ‚Äî every entry in `public-cases.json` has a unique id, a 64-hex sha256, an `https` URL source, a violation class, no unknown keys, and boolean skip flags (guards typos against the unvalidated JSON load).
2. **baseline presence** (always) ‚Äî every non-`dciodvfy_skip` case has a committed baseline JSON.
3. **per case** (only when `RUN_PUBLIC_CONFORMANCE` is set and not `0`/`false`) ‚Äî all cases are prefetched concurrently in a `beforeAll` into the content-addressed cache (sha256 verified), then each non-`dciodvfy_skip` case runs the same four tests as the synthetic and local suites via `differentialSuite.ts`. For `curate_skip` cases the passthrough test instead pins that `curateOne` still rejects the file. `dciodvfy_skip` cases get a fetch/verify test only.

## Public dirty corpus

`public-cases.json` is a local catalog in the same format as dicom-synth's bundled one (loaded with its `loadCasesFromJson`). Each case is a real quirky DICOM file, fetched on demand from upstream and pinned by sha256 ‚Äî an upstream change fails loudly with a "bump metadata" error rather than drifting silently. Fetched files land in `~/.cache/dicom-synth-testcases/<sha256>/` and are reused across runs; CI caches that directory keyed on the catalog hash, so network is only needed when the catalog changes.

All current cases come from the MIT-licensed [pydicom test_files](https://github.com/pydicom/pydicom/tree/main/src/pydicom/data/test_files) set, pinned to the `v3.0.1` tag:

| Case | Quirk | Flags |
|------|-------|-------|
| `pydicom-badVR` | Invalid VR encoding | `dciodvfy_skip` ‚Äî dciodvfy aborts on this file; fetch/verify only |
| `pydicom-CT-small` | Mostly-valid CT with real-world warnings | |
| `pydicom-ExplVR-BigEnd` | Retired Explicit VR Big Endian transfer syntax | |
| `pydicom-image-dfl` | Deflated Explicit VR Little Endian | |
| `pydicom-J2K-pixelrep-mismatch` | J2K codestream vs dataset pixel representation mismatch | |
| `pydicom-J2K-embedded-seq-delim` | Sequence delimiter embedded in encapsulated pixel data | |
| `pydicom-MR-truncated` | Truncated pixel data | |
| `pydicom-SC-rgb-small-odd` | Odd-length pixel data value | |
| `pydicom-nested-priv-SQ` | Private sequences nested in private sequences | |
| `pydicom-no-meta-group-length` | File meta without group length | |
| `pydicom-UN-sequence` | Sequence encoded with VR UN | `curate_skip` ‚Äî dcmjs cannot parse it; passthrough test pins the rejection |
| `pydicom-meta-missing-tsyntax` | File meta without TransferSyntaxUID | `curate_skip` ‚Äî dcmjs cannot parse it; passthrough test pins the rejection |

Each skip flag is a documented coverage gap, recorded in the case's `notes`. `curate_skip` cases assert that `curateOne` still rejects the file ‚Äî if a dcmjs upgrade makes one parseable, the test fails and the flag should be removed. The `update:conformance-baselines` script fetches through the same default cache as the tests, so files are only ever downloaded once per machine.

**Adding a case:** download the file, `shasum -a 256` it, add an entry (prefer a tag-pinned URL over a branch), verify dciodvfy and passthrough curate behaviour locally, set skip flags with a reason in `notes` if needed, then run `pnpm update:conformance-baselines` and commit the new baseline JSON.

### `dciodvfy.local.test.ts`

Runs only when `CONFORMANCE_LOCAL_PATH` is set (resolved inside the suite, not at import). Same four checks per discovered `.dcm` as the synthetic differential suite. Baseline drift and snapshot-pinned passthrough run only when the matching `*.dciodvfy-baseline.json` file exists under `CONFORMANCE_LOCAL_BASELINE_DIR`.

## Allowlist

Some `dciodvfy` warnings are disputed or context-dependent. Regexes in `allowlist.ts` (`CONFORMANCE_ALLOWLIST`) are subtracted in `violationSet` before comparisons.

**Remember:** `dciodvfy` remains a regression aid, not a certification tool.

## dicom3tools (required binary)

Conformance tests invoke **`dciodvfy`** from [dicom3tools](https://www.dclunie.com/dicom3tools.html). It is not shipped with this repo or `dicom-synth`.

```bash
# Debian/Ubuntu
sudo apt-get install dicom3tools
export DCIODVFY_PATH=/path/to/dciodvfy   # optional override
```

**CI:** `.github/workflows/ci.yml` installs `dicom3tools` before `pnpm test:conformance`.

## Local and private fixtures

You can use your own DICOM without forking dicom-curate. Point the conformance Vitest project at files on disk via environment variables.

| Variable | Required | Purpose |
|----------|----------|---------|
| `CONFORMANCE_LOCAL_PATH` | Yes (for local tests) | Path to one `.dcm` file, or a directory tree searched recursively for `*.dcm`. Multiple roots: separate with `path.delimiter` (`:` on Linux/macOS, `;` on Windows). |
| `CONFORMANCE_LOCAL_BASELINE_DIR` | No | Directory where **your team** stores `*.dciodvfy-baseline.json` files (e.g. a private repo or gitignored folder). When set, baseline paths are computed for each fixture; drift and snapshot-pinned passthrough run only after you create the JSON (e.g. via `pnpm update:conformance-baselines`). When unset, only live before/after passthrough comparison runs. Required when refreshing local baselines with the update script. |

**Example ‚Äî quick check (no baselines):**

```bash
export CONFORMANCE_LOCAL_PATH=/data/my-study/instance.dcm
pnpm test:conformance
```

**Example ‚Äî regression with team baselines:**

```bash
export CONFORMANCE_LOCAL_PATH=/data/dicom-corpus
export CONFORMANCE_LOCAL_BASELINE_DIR=/data/dicom-corpus/conformance-baselines
pnpm test:conformance
```

**Refresh baselines** after upgrading `dicom3tools` or changing fixtures (writes JSON under `CONFORMANCE_LOCAL_BASELINE_DIR`, not into dicom-curate):

```bash
DCIODVFY_PATH=/path/to/dciodvfy \
  CONFORMANCE_LOCAL_PATH=/data/dicom-corpus \
  CONFORMANCE_LOCAL_BASELINE_DIR=/data/dicom-corpus/conformance-baselines \
  pnpm update:conformance-baselines
```

Baseline filenames follow fixture ids: `study-001.dcm` ‚Üí `study-001.dciodvfy-baseline.json`; nested files use a slug (`nested/series.dcm` ‚Üí `nested--series.dciodvfy-baseline.json`). Multiple roots in `CONFORMANCE_LOCAL_PATH` prefix ids with each root‚Äôs basename (e.g. `corpus--study-001`) so basenames do not collide across trees.

`dciodvfy.local.test.ts` uses `differentialSuite.ts` like the synthetic tests. Default CI does **not** set these variables.

**Not implemented at this time.** Either materialise files on disk first and use local settings, or use public samples. 

## Commands

```bash
pnpm test:conformance
RUN_PUBLIC_CONFORMANCE=1 pnpm test:conformance

CONFORMANCE_LOCAL_PATH=/path/to/file.dcm pnpm test:conformance
CONFORMANCE_LOCAL_PATH=/path/to/corpus \
  CONFORMANCE_LOCAL_BASELINE_DIR=/path/to/baselines \
  pnpm test:conformance

DCIODVFY_PATH=/path/to/dciodvfy pnpm update:conformance-baselines

# Local baselines only (no network for public pydicom case):
SKIP_PUBLIC_CONFORMANCE_BASELINES=1 \
  CONFORMANCE_LOCAL_PATH=/data/dicom-corpus \
  CONFORMANCE_LOCAL_BASELINE_DIR=/data/dicom-corpus/conformance-baselines \
  pnpm update:conformance-baselines
```

Tests resolve `dciodvfy` via `DCIODVFY_PATH`, then `PATH`, then `.cache-dciodvfy/...` if present. If `DCIODVFY_PATH` points at a missing file, a warning is printed and resolution falls back to `PATH`.

Synthetic DICOM: [`dicom-synth`](https://github.com/clintools/dicom-synth) at test time. Passthrough curate needs `dcmjs` ‚â• 0.51.
